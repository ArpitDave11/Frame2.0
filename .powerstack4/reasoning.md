# Reasoning Traces

## ROOT CAUSE ANALYSIS — Mermaid Diagram Failures (2026-04-02)

### STEP 1: Define the problem
Mermaid diagrams generated by the pipeline intermittently fail to render. "Sometimes" errors — not always. This suggests the issue is non-deterministic, tied to AI output variability hitting gaps in the validation/fix pipeline.

### STEP 2: Pipeline flow and defense layers
```
AI generates diagram → validateMermaidSyntax (3 regex fixes) → fixMermaidWithAI (AI quoting fix) → applyDiagramTheme
```

Defense layers:
1. **Fix 1**: Colon edge labels → pipe syntax — EFFECTIVE (caught 33x)
2. **Fix 2**: Unsafe subgraph IDs with `&/():` — PARTIALLY EFFECTIVE (only catches special chars, not spaces alone)
3. **Fix 3**: Unicode arrows → text — EFFECTIVE for free text (caught 25x), MISSES inside quotes
4. **AI Fix**: Unquoted special chars — EFFECTIVE when triggered (heuristic-based, only fires 2/30 times)
5. **Directive check**: First line starts with valid type — EFFECTIVE but skeleton fallback loses content

### STEP 3: Gap analysis — what's NOT caught

| Gap | Root Cause | Impact | Fix Difficulty |
|-----|-----------|--------|---------------|
| Subgraph space-only IDs | Fix 2 regex requires `&/():` | CRITICAL — parser crash | Easy — broaden regex |
| `--"text"-->` syntax | Fix 1 only handles colon-style | HIGH — parser crash | Easy — add regex |
| %%{init:} in AI output | No stripping of init blocks | CRITICAL — theme conflict or INVALID_DIRECTIVE | Easy — strip before validate |
| Unicode in pipe/node labels | Fix 3 IS global but timing edge cases | LOW — mostly caught | Verify ordering |
| classDef in non-flowchart | Prompt doesn't scope styling by diagram type | MEDIUM — silent ignore | Medium — conditional prompt |
| Emoji in labels | No handling at all | LOW — some renderers OK | Easy — strip or replace |

### STEP 4: Why it's intermittent
The AI doesn't always produce the same errors. Across 30 test diagrams:
- 17% had zero issues
- 83% had at least one uncaught issue
- 27% had CRITICAL uncaught issues (would crash the renderer)

The intermittency comes from:
1. **Subgraph naming variability**: Sometimes the AI uses single-word IDs (works), sometimes multi-word (breaks)
2. **Unicode arrow habit**: The AI inconsistently uses → vs "to" even with explicit instructions
3. **Init block generation**: Only happens when the AI decides to add theme customization (~7% of cases)
4. **Edge label syntax**: The AI switches between pipe, colon, and dash-text syntax non-deterministically

### STEP 5: Recommendations (no code changes per user constraint)

**Priority 1 — Fix 3 critical gaps in validateMermaidSyntax:**
1. Broaden Fix 2 regex to catch ALL subgraph IDs with spaces (not just those with `&/():`)
   - Current: `/^\s*subgraph\s+[^\n"[\]]+[&/():][\w &/():]+$/gm`
   - Proposed: `/^\s*subgraph\s+(\w+\s+\w[^\n"[\]]*)\s*$/gm` (any multi-word ID)
2. Add Fix 4: `--"text"-->` → `-->|"text"|` conversion
3. Add Fix 0 (pre-validation): Strip `%%{init:...}%%` blocks from AI output — let applyDiagramTheme handle theming

**Priority 2 — Improve fixMermaidWithAI trigger:**
4. The heuristic (`/\w+\[[^\]"]*[()\/&:;][^\]"]*\]/`) only fires 2/30 times. Consider always running the AI fix OR broadening the heuristic to catch more patterns.

**Priority 3 — Prompt improvements:**
5. Add conditional styling instructions: "classDef is ONLY for flowchart/graph diagrams. For sequenceDiagram and stateDiagram-v2, do NOT include classDef."
6. Add explicit rule: "Do NOT include %%{init:} blocks — theming is handled automatically."
7. Strengthen the edge label instruction: "Use ONLY pipe syntax `-->|text|`. Never use `--text-->` or `A-->B: text`."

### STEP 6: Conclusion
**Root cause is NOT the templates or diagramConfig.type.** The v7.0.0 templates have clean directives. The root cause is **3 gaps in validateMermaidSyntax's regex coverage** combined with **AI output variability**:
1. Space-only subgraph IDs (6/30 = 20%)
2. %%{init:} blocks (2/30 = 7%)
3. --"text"--> syntax (2/30 = 7%)

Combined probability of at least one critical failure per diagram: ~27%. This matches the user's observation of "sometimes" errors.
