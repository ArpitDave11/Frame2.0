========================================================================
1. PROBLEM STATEMENT
========================================================================
FRAME refines epics through a 6-stage AI pipeline once an epic is loaded
into the planner. But before that — during PI planning — Scrum Masters
size raw epics by hand, eyeballing complexity. The result is sprint
loads that drift from real capacity, with no consistent FRAME-grounded
counter-estimate to flag the worst-sized epics before grooming wastes
sprint time.

BRP (Breakdown & Re-groom Planning) gives planners a capacity-driven
sizing surface: load a GitLab crew, set pod capacity (5 inputs), and
run an Analysis pass that produces a FRAME estimate + a variance band
per epic. Epics in the "re-groom" band hand off to the existing 6-stage
pipeline. Capacity, variance, and roll-ups are derived live from inputs
— never stored — so the data model can't drift from its math.

2. SCOPE OF THIS PRD
========================================================================
This PRD covers ONLY the headless layer of BRP: Phases 1–4 of the
7-phase plan (types, store, AI seam, GitLab service). It produces NO
UI, registers NO tab, and ships NO user-facing change. It establishes
a stable, fully-tested foundation that Phase 5 (components) and Phase 6
(wiring) consume.

A separate PRD will be written for P5 → P6 after this one closes.
P7 (polish + AI-assist) gets its own PRD.

5. USERS & USE CASES (headless layer's eventual consumers)
========================================================================
The headless layer has no direct users. Its consumers are:

Consumer 1: Phase 5 components.
  Read store via selectors. Call pure functions at render time. Dispatch
  actions in response to events. The action signatures defined here
  become the component contract.

Consumer 2: Phase 6 wiring.
  Composes services + store + AI seam into the 5 end-to-end flows
  (Load, Capacity, Estimate, Analyze, Re-groom handoff).

Consumer 3: Phase 7 polish + AI-assist.
  Adds the audit log alongside this layer. The seam (AIEstimator) lets
  Phase 7 swap simulator → real estimator without touching consumers.

6. LOCKED DESIGN DECISIONS
========================================================================
D1.  Pure layer lives at src/domain/brp.ts + src/domain/brp.constants.ts
     — matches FRAME convention (src/domain/types.ts, complexity.ts).
     NOT src/types/ or src/lib/ as p1 names them.

D2.  Three architectural invariants enforced by the type shape (Phase 1):
     a. Epic has NO top-level variance/delta/frameEstimate field. All
        FRAME outputs live inside the nullable Epic.frameResult sub-object.
     b. Pod stores CapacityInputs (5 raw inputs). It does NOT store
        totalCapacity. Total is always computeCapacity(inputs).
     c. VarianceBand is a return type of computeVariance — never a
        field anywhere in the model.

D3.  Pure functions: computeCapacity, computeDelta, computeVariance,
     computePodMetrics. Same input → same output. No Date.now, no random.
     Tests are the gate.

D4.  Store at src/stores/brpStore.ts, patterned on initiativeStore.ts.
     Holds inputs and raw data only. NEVER stores derived data
     (variance, delta, totalCapacity, pod metrics). Reads pure functions
     in tests; components call them at render time (P5).

D5.  AIEstimator interface DEFINED IN src/domain/brp.ts (per p2's
     recommended-option settled here). Implementation in
     src/services/brp/ai/. This lets the store import only the
     interface, never the impl.

D6.  Simulator (src/services/brp/ai/simulatedEstimator.ts) is
     deterministic — same epic → same FrameResult — using a hash of
     epic.id as seed. Confidence moves inversely with breakdown variance.
     Returns AsyncIterable<AnalysisEvent> so the real estimator (P7)
     can stream without changing the interface.

D7.  GitLab service composes existing src/services/gitlab/gitlabClient.ts
     (does NOT bypass it, does NOT add raw fetch calls). Lives at
     src/services/brp/brpGitlabService.ts. Returns BRP-typed results,
     mapping GitLab shapes at the service boundary.

D8.  No new external dependencies in this PRD. Zustand v5, Zod 4,
     Vitest 4 — all already in package.json. Recharts (P5) is documented
     as a future install but NOT added here.

D9.  Branch: feature/brp, worktree at /Users/arpit/Documents/FRAME_DEPLOYED-brp.
     Mergeable to main independently of feature/issue-refinery.

D10. Variance UI label decision (sample wording vs plan wording) is
     deferred to Phase 5. Internal type names (agree | caution | re-groom
     | flagged | pending) are locked here; display labels are a P5 concern.

7. FUNCTIONAL REQUIREMENTS
========================================================================

7.1 — Phase 1 (Pure model & math)
F1.1   src/domain/brp.ts exports: FibonacciPoint (literal union 1|2|3|
       5|8|13|21|40|100), AnalysisStatus, VarianceBand, CapacityInputs,
       CapacityResult, BreakdownItem, ReferenceEpic, GeneratedStory,
       FrameResult, Epic, Pod, Crew, PI, PodMetrics, AnalysisEvent,
       AIEstimator (interface).
F1.2   src/domain/brp.constants.ts exports: FIBONACCI_POINTS (readonly
       array), DEFAULT_SP_PER_RESOURCE = 10, VARIANCE_AGREE_THRESHOLD =
       0.20, VARIANCE_CAUTION_THRESHOLD = 0.50, CONFIDENCE_BUMP_THRESHOLD
       = 0.40.
F1.3   computeCapacity(inputs: CapacityInputs): CapacityResult — formula:
       gross = resources × spPerResource × sprintCount;
       holiday = holidayDays × resources;
       leave = leaveDays;
       total = max(0, gross − holiday − leave).
       Returns full breakdown.
F1.4   computeDelta(epic: Epic): number | null —
       frameResult.frameEstimate − humanEstimate.
       null if either side missing.
F1.5   computeVariance(epic: Epic): VarianceBand — order of checks:
       (1) analysisStatus !== 'done' OR frameResult is null
           → 'flagged' if description thin (heuristic TBD in test),
             otherwise 'pending'.
       (2) computeDelta is null → 'pending'.
       (3) ratio = |delta| / max(human, frame);
           ratio ≤ 0.20 → 'agree';
           ratio ≤ 0.50 → 'caution';
           else → 're-groom'.
       (4) If band would be 'agree' and frameResult.confidence < 0.40
           → bump to 'caution'.
F1.6   computePodMetrics(pod: Pod): PodMetrics — returns totalCapacity,
       humanLoad, frameLoad, balance, avgConfidence, epicCount,
       flaggedCount, reGroomCount. humanLoad and frameLoad EXCLUDE
       flagged epics (regression-guard for p1's stated past bug where
       silent inclusion misrepresented load).

7.2 — Phase 2 (Store)
F2.1   src/stores/brpStore.ts exports useBrpStore (note: lowercase 'b'
       to match initiativeStore casing pattern), with state:
         crews: Crew[]; currentPI: PI | null;
         view: 'portfolio' | 'pod';
         selectedCrewId, selectedPodId, selectedEpicId: string | null;
         collapsedPods: Set<string>;
         reGroomOnlyFilter: boolean;
         openModal: 'capacity' | 'epicPicker' | 'podLoader' | 'metrics' | null;
         modalContext: { podId?: string } | null;
         analysisStatus: 'idle' | 'running' | 'done';
F2.2   Actions, grouped:
         Loading: loadCrew, loadPods, loadEpicsIntoPod
         Capacity: updatePodCapacity
         Estimates: setHumanEstimate
         Analysis: runAnalysis, setEpicAnalysisStatus, setEpicFrameResult
         Navigation: setView, selectCrew, selectPod, selectEpic,
                     togglePodCollapse, setReGroomOnlyFilter,
                     openModalFor, closeModal
         Reset: reset
F2.3   The store NEVER stores derived data. No variance, delta,
       totalCapacity, or pod metrics in state. Verified by reviewer
       at the gate.
F2.4   runAnalysis accepts an AIEstimator (parameter — store imports
       only the interface from src/domain/brp.ts, not the impl). The
       actual estimator instance is supplied by the caller (P6 wiring
       provides estimatorProvider).
F2.5   The store starts empty (crews: []). Mock data lives ONLY in
       test fixtures (brpStore.test.ts) — never inlined in the store.

7.3 — Phase 3 (AI seam)
F3.1   src/domain/brp.ts already exports the AIEstimator interface
       (per D5). Shape:
         interface AIEstimator {
           analyzeEpic(epic: Epic, refs: ReferenceEpic[]): AsyncIterable<AnalysisEvent>;
         }
       AnalysisEvent shape:
         | { kind: 'started'; epicId: string }
         | { kind: 'progress'; epicId: string; pct: number }
         | { kind: 'done'; epicId: string; result: FrameResult }
         | { kind: 'error'; epicId: string; message: string }
F3.2   src/services/brp/ai/simulatedEstimator.ts exports
       createSimulatedEstimator(): AIEstimator. Deterministic: hash
       epic.id → seed → produces a stable FrameResult for that epic.
F3.3   src/services/brp/ai/estimatorProvider.ts exports
       getEstimator(): AIEstimator. Returns the simulator today.
       Single-line swap point for P7.
F3.4   Zod schema for FrameResult lives alongside the estimator
       and validates every event payload. Invalid → error event.
F3.5   No real LLM calls. No Azure config reads. No network. The
       seam must be testable with zero network setup.

7.4 — Phase 4 (GitLab service)
F4.1   src/services/brp/brpGitlabService.ts exports four async
       functions that compose gitlabClient.ts:
         fetchCrews(config): Promise<Crew[]>
         fetchPods(config, crewGroupId: number): Promise<Pod[]>
         fetchPodEpics(config, podSubgroupId: number): Promise<Epic[]>
         fetchReferenceEpics(config, podSubgroupId: number): Promise<ReferenceEpic[]>
F4.2   Each function returns a Result-like shape consistent with
       gitlabClient.ts's existing { success; data?; error? } pattern.
       No exceptions for expected failures; throws only for programmer
       errors.
F4.3   GitLab → BRP mapping happens at this layer (the service is the
       boundary). Returned Epic objects have analysisStatus: 'raw',
       frameResult: null, humanEstimate: null. NEVER pre-filled.
F4.4   fetchReferenceEpics returns closed epics with actual story-point
       totals attached. Filter rule TBD with user before B-11 if not
       obvious from existing gitlabClient.fetchEpics signature.
F4.5   No direct fetch() calls. Composes gitlabClient.* exclusively.

12. RISKS & MITIGATIONS
========================================================================
RK1. The "thin description → flagged" heuristic in F1.5 step (1) is
     underspecified. p1 says "epic description too thin to estimate".
     Mitigation: pick a conservative rule at B-3 (e.g., description
     length < 80 chars OR fewer than 2 sentences) and document it
     in brp.ts as a TODO for P5 to refine with real data. Surface to
     user via AskUserQuestion before locking if it changes pure-function
     behaviour meaningfully.

RK2. ReferenceEpic filtering rule (F4.4) depends on what existing
     gitlabClient.fetchEpics returns. May need a new optional state
     filter param.
     Mitigation: at B-10, read gitlabClient.ts before extending; if a
     simple post-filter on the response works, use it; if it requires
     a client change, escalate via AskUserQuestion (gitlabClient.ts is
     not scope-guarded but is shared and warrants the question).

RK3. Live smoke (B-11) requires the user's GitLab token. May not be
     possible to verify in this session.
     Mitigation: skip-by-default; document manual run command in the
     journal; B-11 verification can be the mocked tests + a recorded
     smoke output from the user later.

RK4. Zustand 5 testing patterns differ from Zustand 4. The IR work
     used Zustand 5 (initiativeStore exists); mirror its test patterns.
     Mitigation: read initiativeStore.test.ts before writing
     brpStore.test.ts (B-5).

RK5. The simulator's confidence calculation (B-9) is non-obvious.
     "Confidence moves inversely with breakdown variance" needs a
     concrete formula.
     Mitigation: define it as 1 − stdev(breakdown.points)/mean(...)
     clamped to [0.1, 0.95]. Document in code. Adjust if tests
     reveal it's unstable on edge cases.

RK6. Hooks (pre-edit-protect-tests.sh) may block edits to brp.test.ts
     once it exists, because the hook treats *all* test files as
     protected after creation.
     Mitigation: per the IR devlog's noted pattern, edits to test files
     in the same task that created them are done via rm + Write. If
     a later task needs to add tests to an earlier task's test file,
     do delete-and-rewrite (the hook accepts this).
