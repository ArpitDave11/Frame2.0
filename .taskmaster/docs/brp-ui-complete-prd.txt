========================================================================
1. PROBLEM STATEMENT
========================================================================
The BRP headless layer (Phases 1–4) is shipped on feature/brp: types,
store, AI seam (simulator), GitLab service. 176 mocked tests + 1 live
smoke passing. But the feature is invisible to users — there is no tab,
no view, no UI. The planner cannot load a crew, set capacity, type
estimates, run Analysis, or see results. Phase 6's wiring isn't there,
so even if a component existed it couldn't talk to GitLab or the
estimator. Phase 7's audit log isn't there — a UBS/banking-regulated
context requires append-only audit traceability for any AI-assisted
decision.

This PRD finishes BRP: components (P5), the wiring that connects them
to the store + services + AI seam (P6), and the polish + AI-assist +
audit log + real LLM swap (P7) that makes it production-ready.

========================================================================
2. SCOPE OF THIS PRD
========================================================================
Phases 5 + 6 + 7 of the 7-phase plan. Everything from "components don't

5. USERS & USE CASES
========================================================================
Primary user: Scrum Master or product engineer running PI planning at
UBS WMA. They have a GitLab root group containing crews → pods →
epics.

Use case 1: "Set up capacity for my pod"
  - User opens BRP tab → Load crew → Select pod → Capacity dialog → enters
    resources, spPerResource, sprintCount, holidayDays, leaveDays → Save.
    Total capacity renders live in the dialog and on the pod row.

Use case 2: "Compare planner estimates with FRAME"
  - User types humanEstimate in the inline column for each epic.
  - Clicks "Run Analysis" → progress bar advances epic-by-epic →
    variance bands render per epic (Agree/Caution/Re-groom/Flagged).

Use case 3: "Re-groom the wide-variance epics"
  - User filters to "Re-groom only" → clicks a re-groom epic → DetailPanel
    opens showing FRAME's breakdown + rationale + references → clicks
    "Send to Refinement Pipeline" → handoff to existing refinePipelineAction.

Use case 4: "Audit who/when an estimate was changed"
  - Compliance officer pulls the audit log → sees every capacity
    update, every humanEstimate edit, every Analysis run with
    timestamps + actor (MSAL'd user) + before/after values.

6. LOCKED DESIGN DECISIONS
========================================================================
D1.  Tab id: 'brp'. Same pattern as IR's 'issueRefinery'.
D2.  Components live at src/components/brp/{...}. Tests colocated.
D3.  Recharts for MetricsModal (variance bar chart + capacity donut).
     ~200KB minified gzip. Locked at session start.
D4.  Action layer at src/actions/brpActions.ts (single file). Mirrors
     the action-boundary pattern of refinePipelineAction.ts and
     refineIssueAction.ts.
D5.  runAnalysis onError callback: brpActions provides a closure that
     calls uiStore.addToast({ type: 'error', title: ... }). The store
     stays UI-agnostic; brpActions is the only file that imports
     uiStore from BRP scope.
D6.  Progress state: add `analysisProgress: { completed, total,
     currentEpicId } | null` to BrpState. runAnalysis updates it via
     a new setProgress action. Progress events from the estimator
     advance pct on the current epic; epic-done events bump completed.
D7.  Result<T> widening: brpGitlabService.ts changes to return
     { success: false, error: { code: string; message: string; cause?: unknown } }.
     Phase 6 callers branch on error.code ('auth' | 'ratelimit' |
     'network' | 'unknown'). Add a unit test asserting the discrimination.
D8.  Real LLM estimator: new src/services/brp/ai/azureOpenAIEstimator.ts.
     Composes existing src/services/ai/azureClient.ts (the same client
     used by the epic pipeline). DOES NOT add a new HTTP client.
     Gated live smoke pattern, same as B-11.
D9.  Audit log: src/services/brp/auditLog.ts + brpStore subscription
     (NOT an action — a side-effect subscriber). Append-only in memory
     today; localStorage persistence is a v2 enhancement (out of scope).
     Captures: action name, timestamp, actor (msal user.id if available,
     else 'unknown'), before/after for state-mutating events.
D10. Design tokens: zero raw hsl() calls, zero font literals like
     'Frutiger'. Use src/theme/tokens.ts (existing). VarianceBadge
     ports from ui_sample_brp but rewrites the styling against tokens.
D11. Single deep-review checkpoint after P6 (B-31), before P7 starts.
     Same 5-agent pattern as the headless deep-review.
D12. CLAUDE.md "Active Work" section is updated AT THE END of this
     PRD (B-35), not mid-flight.

7.2 — Phase 6 wiring (single action file)

F6.1   `src/actions/brpActions.ts` — exports these actions:
         `loadCrewAction(crewGroupId)` — fetchCrews + loadCrew.
         `loadCrewPodsAction(crewId, crewGroupId)` — fetchPods +
            loadPods.
         `loadPodEpicsAction(podId, podSubgroupId)` — fetchPodEpics +
            loadEpicsIntoPod.
         `runAnalysisAction()` — getEstimator + getReferences resolver
            + onError surfaces to uiStore.addToast + abort-on-tab-leave.
         `handoffEpicToRefinementAction(epicId)` — pull Epic from
            brpStore + writeMarkdownIntoEpicStore + refinePipelineAction.
       Each action: reads from brpStore + configStore, calls services,
       writes results back to brpStore. Boundary-pattern compliant —
       components do not call services directly.
F6.2   `BrpProgress` state + setProgress action added to brpStore.
       runAnalysis (in store) now also writes progress on every event.
F6.3   `Result<T>` widened in brpGitlabService.ts:
         `{ success: true; data: T }` |
         `{ success: false; error: { code: 'auth'|'ratelimit'|'network'
            |'unknown'; message: string; cause?: unknown } }`.
       Mock tests updated; live smoke updated to assert error.code on
       a 401 scenario.
F6.4   `runAnalysisAction` consumes the AbortSignal from the user's
       Cancel button (P5 component) AND from a tab-change unsubscribe.
F6.5   `handoffEpicToRefinementAction` writes the epic body into
       `epicStore.setMarkdown(...)` and switches `uiStore.activeTab`
       to `'planner'`, then triggers `refinePipelineAction`. The
       existing 6-stage pipeline handles it from there.

7.3 — Phase 7 polish + AI-assist + audit log + real LLM (~10 files)

F7.1   Design-system audit: every component reviewed; zero raw `hsl()`,
       zero literal font names, all spacing via tokens.
F7.2   Accessibility audit: keyboard nav on all components, ARIA labels,
       no color-only signals (variance bands also carry icons / text).
F7.3   AI-assist: capacity pre-fill.
       New: `src/services/brp/ai/capacityPrefill.ts` — given a
       crewGroupId, fetches member count via gitlabClient + currentPI
       sprints + holidays from a config OR a sensible default. Returns
       `CapacityInputs`. UI hook: "Pre-fill" button in CapacityDialog.
F7.4   AI-assist: unrealistic-capacity nudge.
       `src/services/brp/ai/capacityNudge.ts` — given a CapacityInputs,
       compares against pod history (or a constant baseline if no
       history). Returns a string warning or null. Rendered as inline
       help text in CapacityDialog.
F7.5   AI-assist: variance interpretation text.
       `src/services/brp/ai/varianceInterpreter.ts` — given an Epic,
       returns a short templated string per band. Pure function. No
       LLM. Tested.
F7.6   AI-assist: duplicate-epic detection.
       `src/services/brp/ai/duplicateDetection.ts` — token-overlap
       similarity (Jaccard or shingled) across an array of Epics.
       Returns pairs above threshold. Pure function. Rendered as a
       collapsed "Potential duplicates" section in PortfolioView.
F7.7   Audit log.
       `src/services/brp/auditLog.ts` — append-only in-memory log
       capturing every state-mutating event with timestamp, actor
       (msalAccount?.idTokenClaims?.oid if available, else 'unknown'),
       action name, before/after delta. Exposed as a brpStore
       subscriber (NOT a store action). UI: a hidden "Audit" panel
       toggleable from DetailPanel (collapsed by default). The log
       data structure is exportable as JSON for compliance asks.
F7.8   Real LLM estimator.
       `src/services/brp/ai/azureOpenAIEstimator.ts` — implements
       `AIEstimator`. Composes existing `src/services/ai/azureClient.ts`.
       Streams progress events as the LLM streams. Falls back to
       'error' event on rate limit / auth failures. Schema-validates
       the FrameResult before yielding 'done' (the schemas in
       `src/services/brp/ai/schemas.ts` exist for this).
       Mocked tests + gated live smoke (same pattern as B-11).
F7.9   Estimator provider swap.
       `src/services/brp/ai/estimatorProvider.ts` — `getEstimator()`
       body changed from `createSimulatedEstimator()` to:
         `import { isAIEnabled } from '@/services/ai/aiClient'`
         `return isAIEnabled(useConfigStore.getState().config)`
         `  ? createAzureOpenAIEstimator(config)`
         `  : createSimulatedEstimator()`
       The drop-in-equivalence test in simulatedEstimator.test.ts is
       updated to skip when isAIEnabled returns true; a new test pins
       the AzureOpenAI path.
F7.10  All component empty / loading / error / filtered-empty states
       have real copy (no placeholder text). Tests assert each state.

12. RISKS & MITIGATIONS
========================================================================
RK1. Recharts is large (~200KB minified). Mitigation: ensure it's
     code-split (vite handles this automatically for components
     that import it lazily). Track bundle size via npm run build
     output.

RK2. Real LLM estimator integration may hit Azure rate limits during
     dev. Mitigation: skip-by-default live smoke; mocked tests
     exercise the contract.

RK3. The Result<T> widening in brpGitlabService.ts is an API change.
     Mitigation: it's only consumed by brpActions.ts (new in this PRD).
     The mocked tests get updated alongside. No external consumers.

RK4. Component scope is large (12 components). Mitigation: each
     component is small (<200 lines), single-responsibility, tested.
     Atomic tasks (B-18..B-27) keep commits focused.

RK5. Audit log design may be over-specified for v1 needs.
     Mitigation: keep in-memory only (no persistence). JSON export.
     Phase 5/6 dogfood will reveal what compliance actually needs.

RK6. Real LLM may not respect the Zod schema (LLM emits non-Fibonacci
     frameEstimate, etc.). Mitigation: azureOpenAIEstimator.ts
     schema-validates every event payload before yielding. Invalid
     events become 'error' events with a clear message.

RK7. handoffEpicToRefinementAction writes to epicStore which IR also
     writes to. Mitigation: read-only consumption in handoff; epic
     body goes in, existing pipeline takes over. No conflicts.

RK8. The kit hardening H3 hook (block test-file edits) will be hit
     ~30 times across this PRD. Pattern: `rm + Write` (documented in
     IR devlog).

========================================================================
13. ROLLBACK PLAN
========================================================================
The feature is purely additive at the workspace tab level:
- Rollback option 1: revert the merge commit. The BRP tab disappears,
  existing tabs continue to render.
- Rollback option 2: remove the `'brp'` entry from TabId in uiStore;
  the tab becomes inaccessible but the code stays.
- No data migration. No backend changes. No deployed surface beyond
  the frontend bundle.

The audit log is in-memory only, so rollback loses its contents — but
that's by design for v1; compliance asks for an audit log of a session
that no longer exists are answered by "we don't have it; load again."
